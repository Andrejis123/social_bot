# Social_Bot Workflow & Process Insights

*Based on: 48 commits (2026-05-04 to 2026-07-03), branch structure, hooks in `.claude/`, `justfile`, `CLAUDE.md`, `docs/` + `research/` notes, test history, the 2026-07-03 Claude insights report (109 sessions), and the project journal. Direct by request; evidence cited.*

## 1. What is being done well: keep it

- **The justfile is a real runbook, not decoration.** Every cron entry, deploy step, and manual operation has a recipe with a comment explaining *why* (`cron-posts` explains per-account staggering; the deploy section explains the volume-mount vs baked-in split). Someone new could operate this system from `just --list` alone. This is rarer than it should be.
- **Destructive operations are uniformly dry-run-by-default with apply flags and abort guards.** `purge`, `restore_from_bundle`, `sweep_drive_orphans`, `backfill_storage_paths`, `_cleanup_stale_junk` all follow the same pattern, and the guards are thoughtful (purge aborts on empty candidate set; orphan sweep refuses to treat a failed ledger query as "everything is an orphan"). This discipline plainly came from handling real client data and it shows.
- **Operational learning gets encoded, not just remembered.** The two prod regressions that actually bit (dev-only python-pptx, missing `assets/` COPY) became permanent assertions in `scripts/deploy_check.py`. The git-guard hook physically blocks raw `git commit` and routes through `/commit-session` with review levels. The `MANDATORY`/skill-invocation guardrail in CLAUDE.md exists because a skill step once got faked. Each incident produced a mechanism, not a resolution to be more careful.
- **Observability is built in from the data side:** `run_history` + `run_item_errors` per run, structured logging with bound run context, Telegram notifications with per-job semantics, and `/data-health` as a queryable quality report. For a one-person operation this is a strong feedback loop.
- **Commit messages are informative and honest** ("fix stale schedule", "Revert posts scraper to directUrls (usernames not supported by actor)") and the recent conventional-prefix style (`feat:`/`fix:`/`docs:`) is consistent. Commits bundle a coherent theme rather than drive-by noise.
- **Decision notes live next to the code.** `docs/archive_architecture.md`, the rollout tracker, the Facebook brief written explicitly as "a self-contained handoff to make a future session faster", and `research/` for evaluated-but-deferred options (TikTok, actor market). The journal keeps live state (cron table, active experiments with exact timestamps and drive IDs). This is a working memory system, and the insights report confirms it is actually used across sessions.
- **The `_spike_*.py` convention** is a legitimate lightweight experiments folder: prompts were validated in spikes (`_spike_cluster.py`, `_spike_gemini_synthesis.py`) before productionizing into `reports/synthesis.py`. The problem is not the pattern; it is that spikes are never garbage-collected (see below).

## 2. Where the process is costing time or causing bugs

- **No CI. This is the single biggest process gap.** There is no `.github/`, so `just check` runs only when someone remembers to run it locally, and the git history proves the cost: the repo shipped with a broken lint gate for 7 weeks (`bcc537c` "Make ruff lint gate green", 2026-06-22) and mypy silently broken until 2026-06-29 (`dec61ca`, journal: "150 errors; `just check` halted at mypy and never reached pytest; the old 'ruff + pytest green' claim was masking that"). A 20-line GitHub Actions workflow would have surfaced both the day they regressed. The git-guard/commit-session pipeline reviews *diffs*; CI verifies *state*; you have the first and not the second.
- **Tests arrived seven weeks into the project** (first suite `a3e6d2f`, 2026-06-22), after the period in which most scraper-shape bugs happened (the May commits are a chain of "fix stories path", "fix pagination on pinned posts", "revert posts scraper"). Since then the discipline is real (172 fast tests, bug-fix protocol demanding a failing test first), so this is mostly historical, but the uncovered zones that remain (scraper tiering, retry_ai, renderer helpers; see AUDIT_CODEBASE §6) are exactly the code written before the test habit started.
- **Deploy correctness depends on human memory in two places** the tooling doesn't cover. (a) `uv.lock` is gitignored, so the carefully built `deploy` → `deploy-check` chain verifies your *code* is in the image while dependencies silently float per rebuild. (b) The VPS crontab is untracked server-side state edited over SSH; the journal itself flags it and the "Facebook scraping was never actually off" incident was a config/cron mismatch nobody could have caught from the repo. Both contradict the repo's otherwise-strong "encode it, don't remember it" ethos.
- **Alerting covers failure, not absence.** Telegram fires when a run fails; nothing fires when a run *doesn't happen* (crontab typo, container image broken, VPS down). The HikerAPI credit-drop investigation (journal, 2026-07-03) is an instance of the general problem: external spend and scheduled activity have no automatic watchdog, so anomalies are noticed by eyeball. During the current wind-down this is tolerable; with a paying client it is the first thing that will embarrass you.
- **One-off scripts and demo configs accumulate without a disposal step.** 7 spikes + 4 completed one-off migrations/backfills sit in `scripts/`; 5 of 8 `config/clients/` folders are not live clients; `v1_plan.md` and two PDFs (one 6 MB) sit in the repo root; `research/` files were moved to `docs/` but the deletions are still uncommitted in git status. Each item is small; together they make the repo lie slightly about what is real, and stale configs sit where automation (`DEFAULT_CLIENTS`, `load_client`) can reach them.
- **Docs drift at the edges** even though the big docs are good: `.env.example` missing the Telegram/Drive variables README tells you to fill, README's "3 clients × 6 accounts" vs reality, migration step mentioning only 0001. Onboarding-you-in-6-months follows these instructions and hits errors that cost an hour.
- **Everything is main, push == deploy-intent.** Fine for a solo project at this scale, and the hook's push speed-bump is a reasonable guard. The cost is only latent: no PR surface means the `/code-review`-style tooling and CI gating have nothing to attach to when something risky (schema migration, purge logic) would benefit from a checkpoint. Not urgent; noted so the trade-off stays deliberate.
- From the insights report, on the agent-workflow side: sessions were derailed by environment blockers (macOS TCC revocations, config-dir aliasing) and by agent overreach (expensive sub-agents ignoring model directives, a normal API key over-flagged as an incident). The pattern: agent-driven ops work is high-throughput here, but guardrails arrive *after* each class of incident. Pre-committing budgets/models in skill definitions (as `CLAUDE.md` now does for haiku sub-agents) is the right direction; keep extending it.

## 3. Concrete workflow upgrades, ranked by effort-to-payoff

1. **Commit `uv.lock`; drop the Dockerfile fallback.** ~5 minutes. Makes every deploy reproducible and turns `deploy-check` back into a real guarantee.
2. **Add GitHub Actions running `just check` on push to main.** ~30 minutes (uv has first-class CI setup). Directly prevents a repeat of the 7-week broken-gate episode. Add `pip-audit`/`uv` dependency audit later if desired.
3. **Version the crontab.** Commit `deploy/crontab.txt`, add `just crontab-install` (`ssh ... 'crontab < /opt/social-bot/deploy/crontab.txt'`) and `just crontab-diff` to compare live vs repo. ~1 hour, kills a whole class of "what is actually scheduled?" uncertainty; the journal already wanted this.
4. **Dead-man switch for the cron.** Cheapest version: a daily cron on the VPS that queries `run_history` for expected jobs in the last 24h and Telegrams a summary (or just wire healthchecks.io pings into the recipes). ~1-2 hours. Converts "absence of alerts" from ambiguous to meaningful. Add a weekly HikerAPI-balance line to the same message and the credit-watch memory item automates itself.
5. **`scripts/archive/` sweep + config pruning.** ~30 minutes. Move completed one-offs and dead spikes, delete or quarantine non-live client folders, commit the pending `research/` deletions, drop the PDFs from the root. Do it once; add a line to the session-handoff ritual ("any one-off scripts to archive?") so it stays done.
6. **Drift checks inside `deploy-check`.** It already imports modules and counts assets; extend it to assert `.env` contains every required Settings field (pydantic can do this: instantiate `Settings()` and it fails loudly) and to print the image's installed versions of the 4 critical SDKs. ~1 hour.
7. **Single source for `DEFAULT_CLIENTS`** (a `live_clients.yaml` or a `clients` table query) consumed by both cron drivers. ~30 minutes, removes a guaranteed future drift bug.
8. **One opt-in integration test** (`pytest -m integration`) that runs the real PostgREST queries (embedded joins especially) against the dev Supabase project. ~2 hours. This is the only class of bug the current 2-second suite structurally cannot catch.

## 4. Suggested 30-day plan

**Week 1 — stop the silent failures (items 1, 2, 6).** Commit the lockfile, harden the Dockerfile, stand up CI on main, extend `deploy-check` with the Settings/env assertion. Everything here is mechanical and nothing depends on product decisions.

**Week 2 — make the VPS state visible (items 3, 4).** Version the crontab with install/diff recipes; add the daily run-liveness + HikerAPI-balance Telegram digest. Finish the pending live restore test (`restore_from_bundle agape ... --apply` + report regen) while you are on the box; it is the one unverified leg of the archive cycle and the journal has it flagged.

**Week 3 — pay down the top code findings under the new safety net.** Pagination helper in `db/queries.py` (+ the pagination regression test), `retry_ai` storage-path fix, per-account error isolation in `ingest_posts`, Telegram HTML escaping. Each lands with a failing test first, per your own bug-fix protocol; CI now enforces the suite stays green.

**Week 4 — hygiene and consolidation (items 5, 7, 8).** Repo sweep (spikes, configs, root files, uncommitted deletions), unify `DEFAULT_CLIENTS`, refresh README/.env.example, add the integration smoke test. End the month by re-running `/data-health 30d` and archiving the audit docs' resolved findings into the journal.

Sequencing logic: weeks 1-2 are pure infrastructure with zero product risk and they protect everything after; week 3 touches pipeline code only once CI exists to catch regressions; week 4 is cleanup that is safe to defer but cheap to batch. If a paying client lands mid-plan, weeks 1-2 are the non-negotiable subset.

## If you only do 3 things

1. **Stand up CI running `just check` on every push**: the repo's own history shows weeks-long silent gate breakage that this removes permanently.
2. **Commit `uv.lock` (and the crontab)** so that what runs on the VPS is fully described by the repo, which is the standard the rest of your tooling already aspires to.
3. **Add the daily liveness digest to Telegram** so silence stops being ambiguous: scheduled-jobs-ran + HikerAPI balance in one message.
